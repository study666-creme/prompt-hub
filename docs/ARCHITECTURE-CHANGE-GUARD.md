# 架构改动护栏（给你 + 给 AI）

> **目的**：你不懂代码时，仍能说清「这次改的是表象还是根因」；AI 不能为了修一个问题，悄悄把整站架构改乱（拆东墙补西墙）。  
> 最后更新：**2026-07-02**

---

## 你要记住的一句话

**可以要 AI 修 bug、要更快；若涉及「换一套做法 / 合并两种卡 / 大改加载流程」，AI 必须先说明会坏什么，你再决定做不做。**

---

## 两类改动（AI 必须标明）

| 类型 | 例子 | AI 该怎么做 |
|------|------|-------------|
| **A · 优化 / 修 bug** | 404 只试一次；切断递归；有 `storage://` 的生图卡走和普通卡一样的 batch 签 grid | 可以直接改，改完简短说「动了哪些文件、怎么验收」 |
| **B · 架构调整** | 「生图卡完全当普通卡、删掉 genJobId」；「卡片库改成和社区完全同一套 DOM」；「去掉 Masonry 改虚拟列表」 | **先写风险清单 + 更小方案**，你点头后再动大代码 |

## 为什么不能随便换架构？

Prompt Hub 里**看起来是一个功能**，往往连着多条链：

```text
卡片库缩略图
  → card-gallery（MJ 多图选哪张）
  → supabase-sync（batch 签名 / storage 路径）
  → warehouse-thumb（尚未归档时的补救）
  → card-image-loader（懒加载、404 重试）
  → script.js renderCards（先出 DOM 再拉图）
```

只改其中一环、不说明后果，容易出现：

- 卡片库快了，**社区 Feed 或生图仓库**变慢/跳顶  
- MJ **四宫格和单图槽位**（`jobId#2`）对不上 → 长期缺图  
- 登录后 **云同步 / 归档** 与列表加载打架 → 401 或重复请求  
- 手机 **lazy 加载** 失效 → 只有首屏 4 张有图  

这些在 `docs/AI-PITFALLS.md` 里都有历史案例。

## 真实案例（本仓库）

### ✅ 属于 A · 该做

**现象**：新生图卡 / MJ 慢、不出缩略图；老卡很快。  

**根因**：不是「生图卡天生该慢」，而是  
- 有 `storage://` 仍被送进慢的 `warehouse-thumbs`；  
- MJ 列表 `data-job-id` 丢了 `#2` 槽位。  

**做法**：统一「列表层」——有 storage 就走 batch 签 grid；无 storage 才 warehouse 补救一次；**保留** `genJobId` 给 MJ / 去重 / 修复。  

**风险**：低；不删字段、不改保存格式。

### ⚠️ 属于 B · 要先讨论

**说法**：「生图卡直接当普通卡保存，不要 genJobId 了。」  

**利好**：列表逻辑更简单。  

**风险**（AI 必须先讲）：  
- MJ 多图、变体按钮、按任务重下、recover-warehouse 都靠 `genJobId`；  
- 归档未完成时只有 job、没有 `storage://`，仍需 Worker 补救；  
- 一次性删字段 = **历史卡片和扩展 API 可能全断**。  

**更小方案**：保存时尽量写成 `storage://`（和普通卡一样显示）；**元数据保留** `genJobId`——这正是 20260701n 的方向。

### ❌ 拆东墙补西墙（禁止）

- 为了卡片库首屏，**await 整页 prefetch 才插 DOM** → 首屏更慢（已踩过）  
- 为了修 404，**列表里 open backfill + warehouse 互调** → 栈溢出、卡死（已踩过）  
- 为了少代码，**社区和卡片库共用一套会清空列的布局** → 滚动跳顶、骨架盖掉真卡

## 你提需求时，可以这样写（复制即用）

```text
先别大改架构。请说明：
1. 这是 A 优化还是 B 架构调整？
2. 可能坏哪些已有功能（社区/生图/MJ/手机/同步）？
3. 有没有更小改法？
确认后你再改代码。
```

---

## AI 改完必须给你的验收（加载类）

1. 硬刷新后 `window.__APP_BUILD__` 是否为当前构建号  
2. 控制台是否无 `Maximum call stack size exceeded`  
3. 卡片库 / 社区各滑一页，Network 是否没有签名请求风暴  
4. 若有 MJ 卡：是否能在 1～3 秒内出缩略图（视网络而定）
